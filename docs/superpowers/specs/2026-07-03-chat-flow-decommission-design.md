# Отключение чат-флоу воронки (Mini App — единственный интерфейс) — design spec

Дата: 2026-07-03
Статус: approved

## Контекст

Подпроекты 2a (квиз) и 2b (детали продукта/оплата/консультация) сдали в Mini App полный аналог чат-воронки. Ещё в спеке 2a это было явно зафиксировано как отложенный последний шаг: «Полное отключение чат-воронки (переключение `/start` на кнопку «Открыть Mini App» без дублирующего чат-флоу) — маленькая отдельная задача в конце подпроекта 2b». Пользователь явно подтвердил объём: отключается **только** пользовательский квиз/покупка/консультация-флоу (Блоки 0–9 ТЗ), административные команды в чате (`/settings`, `/export_leads`, `/redeliver`) остаются работать как есть — другого способа их вызвать пока нет (админ-экран в Mini App — отдельный ещё не начатый подпроект 3).

Bot уже настроен с постоянной Menu Button (`MenuButtonWebApp`, `asgi.py::_bot_lifecycle`), открывающей Mini App рядом с полем ввода — эта кнопка не трогается.

## Что удаляется полностью

Файлы, реализующие Блоки 0–9 ТЗ через inline-кнопки чата:
- `handlers/start.py` (Блок 0, приветствие, retake)
- `handlers/consent.py` (Блок 1)
- `handlers/test.py` (Блоки 2–4, тест и результат)
- `handlers/menu.py` (Блоки 5/9, вход в продукт и умное меню)
- `handlers/practicum.py` (Блок 6)
- `handlers/book.py` (Блок 8)
- `handlers/consult.py` (Блок 7)

Удаляются без замены — коммит в git сохраняет историю, восстановить при необходимости можно оттуда; оставлять неиспользуемый код «на всякий случай» не в стиле проекта (см. существующие правила по избеганию мёртвого кода).

## Что меняется

### `bot.py::build_dispatcher`
Убираются `include_router` для семи удаляемых модулей. Остаются: `admin`, `settings_admin`, `text_input` (в этом порядке, `text_input` — последним, как и сейчас).

### `handlers/text_input.py`
Единственный catch-all для свободного текста. Убирается ветка диспетчеризации к `consult.handle_email_input` (модуль удалён) — остаётся только диспетчеризация к `settings_admin.handle_setting_input` (ввод значений `/settings`). Докстринг файла обновляется: больше не «сбор email + значения настроек», а просто «значения настроек».

### `handlers/start.py` — новая замена
Вместо удаляемого файла — новый минимальный `handlers/start.py`, только `/start`:
```
Привет! Диагностика «Какой нейрокод блокирует твой доход» открывается кнопкой «Открыть» рядом с полем ввода 👇
```
Без клавиатуры (Menu Button уже видна пользователю сама по себе). Без retake-логики, без блока приветствия M0.1-M0.3 — всё это теперь живёт в Mini App (`WelcomeCarousel`).

### `scheduler.py` — напоминания R1-R6
Логика отбора получателей (`due_reminder_users`, `_CANCEL_CHECK`, дедупликация через `reminders_sent`) не меняется — чекпоинты (`AWAITING_CONSENT`, `IN_TEST`, `RESULT_SHOWN`, `PRACTICUM_VIEWED`, `CONSULT_VIEWED`, `BOOK_VIEWED`) читаются и пишутся Mini App'ом идентично чату, отбор остаётся корректным без изменений.

Меняется только клавиатура: `_REMINDER_KB` (6 разных builder'ов с callback-кнопками в удалённые хендлеры) заменяется одной общей клавиатурой `open_miniapp_kb(url)` — единственная кнопка «Открыть» с `web_app=WebAppInfo(url=...)`, открывающая Mini App напрямую. Mini App сам покажет нужный экран по вернувшемуся чекпоинту (`resolveScreen` уже это делает для всех шести чекпоинтов, задействованных в R1-R6) — так что единая кнопка для всех шести кодов корректна и ничего не теряет по сравнению со старыми специфичными кнопками.

### `keyboards/inline.py`
Удаляются билдеры, используемые только удаляемыми хендлерами: `next_kb`, `consent_kb`, `question_kb`, `result_next_kb`, `offer_kb`, `practicum_intro_kb`, `practicum_buy_kb`, `book_intro_kb`, `book_buy_kb`, `consult_intro_kb`, `smart_menu_kb`, `retake_kb`, `reminder_cta_kb`, приватные хелперы `_QUESTION_LABELS` и `_menu_labels`.

**`after_product_kb` — сигнатура меняется, не просто остаётся.** Сейчас она строит персональные кросс-ссылки вида «Посмотреть практикум» с `callback_data=f"offer:{p}"` — этот callback обрабатывал `handlers/menu.py::open_product`, который тоже удаляется. Оставить эти кнопки как есть — значит после деплоя в сообщении о выдаче доступа (`payments/delivery.py`, шлётся независимо от интерфейса покупки) будет мёртвая кнопка. Новая версия: если есть ещё непроданные продукты — одна кнопка «Посмотреть другие варианты» (`web_app`, открывает Mini App — там актуальное умное меню M9); если непроданного не осталось — пустая клавиатура, как и сейчас. Сигнатура получает третий параметр `miniapp_url: str`:
```python
def after_product_kb(current: str, available: list[str], miniapp_url: str) -> InlineKeyboardMarkup
```
Из-за этого `_other_products_rows` и `_AFTER_LABELS` (использовались только старой версией `after_product_kb` и уже удаляемыми билдерами) тоже удаляются — больше нет персональных лейблов на продукт, только один общий текст кнопки.

**Остаются без изменений**: `_kb` (общий хелпер), `payment_link_kb` (используется `payments/delivery.py` для ссылки на видео практикума).

**`payments/delivery.py`** — оба места, где сейчас вызывается `after_product_kb(product, available)`, обновляются до `after_product_kb(product, available, config.webhook_base_url)` (`config` там уже есть параметром функции).

**Добавляется**: `open_miniapp_kb(url: str) -> InlineKeyboardMarkup` — одна кнопка `web_app=WebAppInfo(url=url)`, текст «Открыть».

### `texts/messages.py`
Из `TEXTS` удаляются ключи `M0.1`–`M0.3`, `M1.1`, `RETAKE_PROMPT`, `M4.A`/`M4.B`/`M4.V`, `M5.A`/`M5.B`/`M5.V`, `M6.1`/`M6.2`, `M7.1`/`M7.2`, `CONSULT_EMAIL_PROMPT`/`CONSULT_EMAIL_INVALID`, `M8.1`/`M8.2`, `M9`/`M9_EMPTY` — использовались только удаляемыми хендлерами.

**Остаются**: `M6.3`, `M8.3` (шлёт `payments/delivery.py` после успешной оплаты — независимо от интерфейса, где куплено), `R1`–`R6` (напоминания, `scheduler.py`).

Модуль-level `QUESTIONS: dict[int, str]` и `RESULT_LABELS: dict[str, str]` удаляются целиком — использовались только в `handlers/test.py`/`handlers/start.py`, больше нигде.

## Что НЕ меняется

- `handlers/admin.py` (`/export_leads`, `/redeliver`, `get_file_id`, `get_forwarded_chat_id`) — без изменений.
- `handlers/settings_admin.py` (`/settings`) — без изменений.
- `payments/` (`webhook.py`, `delivery.py`, `yookassa_client.py`) — без изменений в бизнес-логике; продолжает читать/писать те же чекпоинты и те же `TEXTS["M6.3"]`/`TEXTS["M8.3"]`. Единственная правка — два call site `after_product_kb(...)` в `delivery.py` (см. выше, из-за смены сигнатуры).
- `services/checkpoints.py`, `services/catalog.py`, `services/scoring.py`, `services/settings.py`, `services/validation.py` — без изменений (используются Mini App API как есть).
- `middlewares.py::ActivityMiddleware` — без изменений, продолжает работать для оставшихся команд.
- `api/routers/funnel.py` и весь frontend — без изменений, уже полностью самодостаточны.
- Уже находящиеся «в процессе» пользователи — без миграции: чекпоинт читается Mini App идентично, `/start` направит их открыть кнопку, `resolveScreen` подхватит их с того места, где остановились.

## Явно вне скоупа

- Админ-экран в Mini App (настройки + лиды) — отдельный, ещё не начатый подпроект 3. `/settings`/`/export_leads`/`/redeliver` в чате остаются единственным способом администрирования.
- Изменение бизнес-логики самой воронки — переносится 1-в-1, меняется только точка входа (кнопка вместо текста-дублёра).

## Верификация

- `pytest -v` — регресс всего проекта (61 нетронутый тест из `tests/test_funnel_api.py`/`test_scoring.py`/`test_settings.py`/`test_telegram_auth.py`/`test_validation.py`/`test_models.py` не затрагиваются удалением chat-хендлеров, у которых своих тестов не было).
- Ручная проверка после деплоя: `/start` в чате → короткий текст без кнопок; кнопка Menu Button открывает Mini App и корректно показывает экран по текущему чекпоинту тестового аккаунта; `/settings` и `/export_leads` продолжают работать; временно уменьшить `reminder_delay_hours` через `/settings`, дождаться напоминания — убедиться что текст R-кода приходит с единственной кнопкой «Открыть», ведущей в Mini App.
