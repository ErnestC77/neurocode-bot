# «Назад / другие варианты» на экранах покупки — design spec

Дата: 2026-07-09
Статус: approved

## Контекст

`ProductDetail.tsx` (практикум/книга) и `ConsultDetail.tsx` (консультация) — тупиковые
экраны: попав туда с `Offer.tsx`, пользователь может только купить/записаться, пути назад
нет. Это чекпоинты `practicum_viewed`/`book_viewed`/`consult_viewed` в БД — источник
истины для `resolveScreen`, поэтому чисто фронтовая кнопка «назад» (без похода в API)
не работает: после перезапуска Mini App пользователя снова кинет на тот же экран.

## Backend: новый эндпоинт `POST /api/funnel/back`

В `api/routers/funnel.py`, рядом с остальными эндпоинтами воронки. Если текущий
`checkpoint` пользователя — один из `practicum_viewed` / `book_viewed` / `consult_viewed`,
переводит его в `idle` (`services/checkpoints.py::IDLE`). Из любого другого чекпоинта —
no-op (эндпоинт не предназначен для вызова не по месту, например из `awaiting_email`).
Возвращает `FunnelStateOut`, как остальные эндпоинты.

`idle` — уже существующий чекпоинт: в него переходит пользователь после успешной оплаты
одного из продуктов (`payments/delivery.py`), и именно на нём `Offer.tsx` показывает
короткий текст M9 «с чего ещё начать» вместо длинного вступления блока 5
(`checkpoint === "offer_shown"` → `OFFER_INTRO_TEXTS`, иначе → `M9_TEXT`). Кнопка «назад»
просто досрочно включает уже реализованный путь «умного меню» — правок в `Offer.tsx`
и `resolveScreen.ts` не требуется.

Побочный эффект (желательный): `idle` не входит в `REMINDER_CODES`
(`services/checkpoints.py`), в отличие от `practicum_viewed`/`book_viewed`/`consult_viewed`.
Пользователь, осознанно ушедший смотреть другие варианты, перестаёт попадать под
напоминание «ты смотрел практикум, не забыл?» — что корректно, он не завис, а продолжил
выбор.

## Frontend

- `api/client.ts`: `api.goBack()` → `POST /api/funnel/back`.
- `content/texts.ts`: новая константа `BACK_TO_OFFER_LABEL = "← Другие варианты"`.
- `App.tsx`: `onBack={() => runAction(api.goBack)}` пробрасывается в `ProductDetail`
  и `ConsultDetail`.
- `ProductDetail.tsx`: новый проп `onBack: () => void`. Кнопка рендерится только когда
  `!waiting` — во время «Проверяем оплату…» (после клика «Купить», в ожидании вебхука)
  кнопка скрыта, чтобы не увести пользователя, уже начавшего платить. `waiting` сбрасывается
  только по таймауту/ручной проверке/наступлению оплаты — специально ничего менять не нужно,
  кнопка «назад» просто не участвует в этой ветке рендера.
- `ConsultDetail.tsx`: тот же проп, кнопка показывается всегда (здесь нет состояния ожидания).

## Явно вне скоупа

- `ConsultEmailInput.tsx` — не тупиковый экран в том же смысле (пользователь уже
  подтвердил намерение записаться), кнопка назад туда не добавляется.
- Отмена уже созданной записи `Purchase` при нажатии «назад» во время ожидания оплаты —
  недостижимо, т.к. кнопка скрыта в состоянии `waiting`; если пользователь закрыл оплату
  без «назад», это тот же уже принятый в спеке 2b случай «закрыл ЮKassa без оплаты»
  (см. `2026-07-03-miniapp-funnel-purchase-design.md`, «явно вне скоупа»).

## Верификация

- Backend HTTP-тест (`tests/test_funnel_api.py`, по образцу): из каждого
  `practicum_viewed`/`book_viewed`/`consult_viewed` → `POST /back` переводит в `idle`;
  из постороннего чекпоинта (например `awaiting_email`) — checkpoint не меняется.
- `resolveScreen.test.ts`: новых веток нет (путь `idle → offer` уже покрыт).
- Ручная проверка в реальном Mini App: Offer → Практикум → «Другие варианты» → Offer
  (текст M9, практикум снова в списке) → Книга → «Другие варианты» → Offer; кнопка
  «Купить» → скрывает «назад» на время ожидания оплаты.
