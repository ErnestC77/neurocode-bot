# Mini App — экраны воронки, квиз (подпроект 2a из 3) — design spec

Дата: 2026-07-03
Статус: approved

## Контекст

Продолжение переноса бота «Диагностика нейрокода» в Telegram Mini App. Подпроект 1 (инфраструктура: FastAPI + React/TS/Vite + Tailwind + `@telegram-apps/telegram-ui` + auth через initData) сдан и работает в проде — см. `docs/superpowers/specs/2026-07-03-miniapp-infrastructure-design.md`.

Большая задача «перенести всю воронку в Mini App» разбита на 3 независимых подпроекта. Этот документ — **подпроект 2, часть a**: экраны от приветствия до предложения трёх продуктов (welcome → согласие → 7 вопросов → результат → оффер), **без оплаты**. Оплата и доставка доступа (практикум/книга/консультация) — отдельный подпроект **2b**, потому что редирект на внешнюю страницу оплаты ЮKassa и определение момента возврата в webview — самостоятельная техническая задача, которая не должна тормозить экраны квиза. Админ-экран (настройки + лиды) — подпроект **3**, независимо от 2a/2b.

Полное ТЗ с текстами всех сообщений — `C:\Users\mccaq\Desktop\LOGIC.MD`, Блоки 0–5 (M0.1–M5.V) плюс R1–R3 (напоминания, срабатывающие на этих экранах).

## Визуальный стиль

Определён через мокапы в брейнсторм-компаньоне (пользователь выбрал вариант B): кольцо-прогресс наверху (`N/7` в центре кольца), крупная центрированная карточка вопроса, три золотые pill-кнопки в ряд под ней. Этот же визуальный язык расширяется на остальные экраны 2a:
- **Result** — вместо кольца-прогресса (тест уже пройден) вверху бейдж/иконка с названием кода («Выживание» / «Самозванец» / «Чужие цели»), длинный текст расшифровки разбит на смысловые карточки-блоки.
- **Offer** — 3 карточки продуктов (книга/практикум/консультация) в стиле `@telegram-apps/telegram-ui` cell-list + Tailwind, отфильтрованные «умным меню».
- Палитра неизменна: фон `#162a48`, кнопки/акценты `#e8c96a`, текст на фоне — белый.

## Архитектура: backend остаётся источником истины о прогрессе

Не переносим FSM-логику на клиент. Единственный источник состояния воронки — `users.checkpoint`, как и сейчас (без изменений в `db/models.py`). Frontend не хранит собственную копию правил переходов («после Q7 — подсчёт результата», «без согласия тест не начать») — он читает состояние с бэкенда и рендерит экран, соответствующий вернувшемуся `checkpoint`.

Причина: если бы frontend сам вёл wizard-состояние в памяти и отправлял всё одним запросом в конце, закрытие Mini App на середине теста теряло бы прогресс — прямое нарушение принципа ТЗ «состояние сохраняется на каждом шаге», ради которого изначально спроектирован `checkpoint`.

### Новый роутер: `api/routers/funnel.py`

Пять `POST`-действий зеркалят пять существующих aiogram callback-хендлеров один-в-один (та же бизнес-логика, тот же вызов `db/crud.py` / `services/scoring.py` / `services/catalog.py` — только из FastAPI вместо aiogram):

| Endpoint | Аналог в чате | Действие |
|---|---|---|
| `GET /api/funnel/state` | — (вычисляется заново на каждый `/start`) | Возвращает `FunnelState` |
| `POST /api/funnel/welcome/complete` | `welcome:4` | `checkpoint → AWAITING_CONSENT` |
| `POST /api/funnel/consent` | `consent:accept` | `set_consent()` + `checkpoint → IN_TEST` |
| `POST /api/funnel/answers` `{question_no, score}` | `test:a:{q}:{s}` | `upsert_answer()`; на `question_no == 7` — `compute_result()` + `set_result()` + `checkpoint → RESULT_SHOWN` |
| `POST /api/funnel/offer/show` | `result:next` | `checkpoint → OFFER_SHOWN`, считает `available_products` |
| `POST /api/funnel/retake` | `retake:confirm` | `reset_test()` + `checkpoint → IN_TEST` |

`GET /api/funnel/state`:
```python
class AnswerOut(BaseModel):
    question_no: int
    score: int

class FunnelStateOut(BaseModel):
    checkpoint: str
    consent_given: bool
    result_type: str | None
    answers: list[AnswerOut]
    available_products: list[str] | None  # непусто только когда result_type задан
```

Все пять `POST`-эндпоинтов возвращают тот же `FunnelStateOut` после применения изменения — frontend всегда работает с одной и той же формой ответа и просто перерисовывается.

**Валидация ответа на вопрос** — тот же guard, что уже есть в `handlers/test.py::answer_question`: если `question_no != await crud.next_question_no(tg_id)` (двойной клик, устаревший экран), запрос молча ничего не делает и возвращает текущий (неизменившийся) `FunnelStateOut` — не 4xx, ровно как чат тихо игнорирует повторный callback через `callback.answer()` без действия.

### Auth и активность

`current_client` (из подпроекта 1, `api/deps.py`) дополнительно вызывает `crud.touch_activity(tg_id, username, first_name)` на каждый запрос — прямой аналог `middlewares.py::ActivityMiddleware`, который в чате бампает `last_activity_at` на каждый update. Без этого открытие Mini App не сбрасывало бы таймер напоминаний R1-R6, и пользователь получал бы напоминание, даже активно находясь в приложении.

### Данные

Никаких изменений в `db/models.py` не требуется — переиспользуются существующие таблицы `users`, `answers` и существующие функции `db/crud.py`. `services/scoring.py::compute_result` и `services/catalog.py::get_available_products` переиспользуются как есть (чистые функции/переиспользуемые async-функции, уже не завязаны на aiogram).

## Экраны (`frontend/src/screens/`)

| Компонент | ТЗ | Checkpoint | Комментарий |
|---|---|---|---|
| `WelcomeCarousel.tsx` | M0.1→M0.2→M0.3 | не персистится (как и в чате — welcome не имеет своего checkpoint) | Один компонент, локальный `step` 1..3, кнопка «Далее»/«Как проходить тест?»/«Продолжить», без свайпа. По завершении шага 3 → `POST /api/funnel/welcome/complete` → переход на Consent. |
| `Consent.tsx` | M1.1 | `AWAITING_CONSENT` | Кнопка «Поделиться контактом и начать» → `POST /api/funnel/consent`. Как и в чате — никакого реального Telegram-контакта не запрашивается, кнопка лишь фиксирует `consent_given_at`. |
| `Quiz.tsx` | M2.1–M2.7 | `IN_TEST` | Один компонент на все 7 вопросов; номер вопроса = `state.answers.length + 1`. Кольцо-прогресс `N/7`, вопрос по центру, 3 pill-кнопки (Да/Иногда/Нет) → `POST /api/funnel/answers`. |
| `Result.tsx` | M4.A / M4.B / M4.V | `RESULT_SHOWN` | Текст выбирается по `result_type`. Кнопка «Какой шаг мне делать дальше?» → `POST /api/funnel/offer/show`. |
| `Offer.tsx` | M5.A / M5.B / M5.V, M9_EMPTY | `OFFER_SHOWN` | 3 карточки, отфильтрованные `available_products` (если пусто — текст M9_EMPTY). Плюс кнопка «Пройти тест заново» (см. отступление №1 ниже). |

Копирайт ТЗ переносится как строковые константы прямо в TSX (`frontend/src/content/texts.ts` — по одному экспорту на смысловой блок, аналогично `texts/messages.py`, но независимая копия). Осознанное дублирование: заводить API ради отдачи статичного текста, который не редактируется через будущую админ-панель (та работает с `bot_settings`, не с копирайтом), было бы избыточно для YAGNI.

### Роутинг

`FunnelGate` (в `App.tsx`, заменяет текущий тестовый экран из подпроекта 1) при монтировании делает `GET /api/funnel/state`, затем рендерит компонент по правилу:

```
checkpoint == "new"               → WelcomeCarousel
checkpoint == "awaiting_consent"  → Consent
checkpoint == "in_test"           → Quiz
checkpoint == "result_shown"      → Result
checkpoint == "offer_shown"       → Offer
```

Hash в URL (`#/quiz`, `#/result` и т.д.) синхронизируется С состоянием после каждого успешного действия — для читаемости и чтобы `react-router-dom` (уже установлен в подпроекте 1) участвовал не декоративно. Но при загрузке/reload именно **серверный checkpoint решает**, что рендерить — ручной переход на `#/result` до завершения теста откатится к настоящему экрану, а не покажет чужой результат.

## Осознанные отступления от буквального ТЗ

1. **Вход при уже пройденном тесте.** В чате `/start` при существующем `result_type` показывает `RETAKE_PROMPT` с явным подтверждением («да/нет — пройти заново?»). В Mini App `FunnelGate` при `result_type != null` сразу открывает `Offer.tsx` с этим результатом (он самодостаточен), а «Пройти тест заново» — обычная вторая кнопка на этом экране без диалога-подтверждения → `POST /api/funnel/retake`. Более естественно для self-serve интерфейса.
2. **Карточки продуктов на Offer ведут в заглушку.** Реальная покупка — предмет подпроекта 2b. До его сдачи карточки регистрируются под будущий роут `#/product/:id`, которого ещё не существует (тот же паттерн, что и в подпроекте 1: задеплоен только тестовый экран, реальные экраны появились позже). Клик по карточке в 2a просто не имеет обработчика — не крash, а no-op; сама карточка (название, цена, текст) уже видна и проверяема.
3. **Чат-флоу не отключается в этом подпроекте.** Обе точки входа (aiogram-хендлеры и Mini App) продолжают писать/читать один и тот же `checkpoint` в одной и той же БД — рассинхрона нет, так как ни один из интерфейсов не хранит скрытого локального состояния. Полное отключение чат-воронки (переключение `/start` на кнопку «Открыть Mini App» без дублирующего чат-флоу) — маленькая отдельная задача в конце подпроекта 2b, когда там будет собрана покупка целиком.

## Явно вне скоупа

- Оплата ЮKassa, доставка практикума/книги/канала, «умное меню» M9 как отдельный экран после покупки — подпроект 2b.
- Экран консультации (M7.1/M7.2, сбор email) — подпроект 2b (относится к покупке/лидогенерации, не к квизу).
- Админ-экран (настройки + лиды) — подпроект 3.
- Отключение чат-воронки — конец подпроекта 2b.

## Верификация

- Юнит-тесты (`tests/test_funnel_api.py`, `fastapi.testclient.TestClient`, как в `tests/test_api.py` подпроекта 1): для каждого из 5 `POST`-эндпоинтов — валидный initData меняет `checkpoint` и возвращает ожидаемый `FunnelStateOut`; невалидный/отсутствующий initData → 401 (переиспользуется существующий паттерн из `tests/test_api.py`); повторный ответ на уже отвеченный вопрос — no-op, а не ошибка; ответ на Q7 — результат вычисляется и совпадает с `services/scoring.py::compute_result` на тех же данных (та же чистая функция, что покрыта `tests/test_scoring.py`).
- Ручная проверка: пройти весь путь welcome → согласие → 7 вопросов → результат → оффер в реальном Mini App через Menu Button, для каждого из трёх результатов (нужно набрать ответы, дающие каждый тип) — визуально свериться с текстом ТЗ и с выбранным стилем (кольцо-прогресс, pill-кнопки, карточки расшифровки). Закрыть приложение на середине теста и открыть заново — прогресс должен сохраниться (та же регрессия, что уже проверялась для чата в исходном плане бота).
- Регресс: чат-версия воронки (`/start`, весь путь через inline-кнопки) продолжает работать без изменений — ручная проверка после деплоя.
