# Telegram-бот «Диагностика нейрокода»

Маркетинговая воронка: 7-вопросный диагностический тест → один из трёх результатов
→ продажа книги (990 ₽) / практикума (2990 ₽) через ЮKassa или бесплатная запись
на консультацию. Полное ТЗ: `new_bot.md` (не входит в репозиторий, хранится у
владельца бота).

Архитектура намеренно зеркалирует соседний проект `allpay-sub-bot`: aiogram3 +
long polling, SQLAlchemy 2.0 async без Alembic, `while True`-scheduler вместо
APScheduler. Ключевое отличие: **никакого aiogram FSM** — весь прогресс живёт
в таблице `users.checkpoint` (Postgres), поэтому бот переживает рестарт
процесса без потери состояния пользователя. Подробности — в комментариях
`services/checkpoints.py` и `scheduler.py`.

## Запуск локально

```bash
python -m venv .venv
.venv\Scripts\activate            # Windows
pip install -r requirements-dev.txt
copy .env.example .env            # заполнить BOT_TOKEN, DATABASE_URL и т.д.
python bot.py
```

Нужен доступный PostgreSQL (`DATABASE_URL`). Таблицы создаются автоматически
при старте (`db/database.py::init_db`).

## Тесты

```bash
pytest
```

Юнит-тестами покрыт только `services/scoring.py` (детерминированный подсчёт
результата теста, включая все случаи ничьей) — единственный чистый модуль в
проекте. Остальное проверяется вручную диалогом в Telegram (см. ниже).

## Переменные окружения

См. `.env.example`. Это только секреты и то, что нужно до первого запуска.
Бизнес-настройки — командой `/settings` в самом боте (доступна только
`OWNER_CHAT_ID` или тому, кто назначен доп. владельцем через саму панель, см.
ниже).

- **`OWNER_CHAT_ID`** — постоянный запасной админ. Доступ к `/settings` и
  `/export_leads` даётся ЛИБО ему, ЛИБО текущему БД-значению `owner_chat_id`
  из `/settings` (какое бы из них ни совпало) — так опечатка в панели не
  может заблокировать доступ самому себе (см. `services/settings.py`).

## Настройки бота (`/settings`)

Редактируются прямо в Telegram, без передеплоя: `book_file_id`,
`practicum_channel_id`, цены книги/практикума, интервалы напоминаний
(`reminder_delay_hours`, `reminder_check_interval`), доп. владелец
(`owner_chat_id`), `yookassa_shop_id`. Хранятся в таблице `bot_settings`
(Postgres), реестр с дефолтами и валидацией — `services/settings.py`.

- **`book_file_id`** — получить: прислать PDF книги боту, он ответит
  `file_id` (`handlers/admin.py::get_file_id`), скопировать в `/settings`.
- **`practicum_channel_id`** — ID закрытого канала практикума (`-100...`);
  бот должен быть администратором канала с правом приглашать пользователей.

## Лиды на консультацию

Лиды хранятся в нашем Postgres (таблица `leads`, модель `db/models.py::Lead`),
на своём сервере — это и есть основное «хранилище», выгружать никуда во внешнюю
систему по умолчанию не нужно.

- **Сбор email**: после нажатия «Записаться на консультацию» (`consult:book`)
  бот переводит пользователя в чекпоинт `AWAITING_EMAIL` и просит прислать
  email обычным текстом — единственное место в боте, где ожидается свободный
  ввод (см. `handlers/consult.py`). Простая валидация формата, при ошибке —
  повторный запрос. Лид создаётся только после получения корректного email.
- **Экспорт**: команда `/export_leads` (только для `OWNER_CHAT_ID`) выгружает
  все лиды из Postgres в CSV (`tg_id, username, email, created_at`) и
  присылает файл прямо в Telegram (`handlers/admin.py`). Если понадобится
  другой формат выгрузки (HTTP-эндпоинт, Google Sheets и т.п.) — потребуется
  отдельная реализация, текущий вариант выбран как самый простой при уже
  Telegram-центричной архитектуре бота.
- Уведомление владельцу о каждой новой заявке (`exports/notifier.py::notify_lead`)
  по-прежнему приходит сразу в `OWNER_CHAT_ID`, теперь с email в тексте.

## Открытые вопросы (см. также план в `.claude/plans/`)

1. **Фискализация ЮKassa (54-ФЗ)**: `create_payment` для книги/практикума
   всё ещё вызывается без `receipt` (email на этапе оплаты не собираем — это
   отдельно от email консультации выше). Если у магазина включена
   обязательная фискализация, платежи могут отклоняться — тогда потребуется
   аналогичный шаг сбора email перед кнопкой «Купить».
2. Render: нужен платный (starter) план — free засыпает через ~15 минут
   бездействия и останавливает long polling.
3. Для `AWAITING_EMAIL` не заведено отдельное напоминание (в ТЗ нет такого
   R-кода): если пользователь нажал «Записаться», но не прислал email и ушёл —
   он не получит напоминание вернуться. Можно добавить по аналогии с R1-R6,
   если понадобится.

## Верификация вручную

1. `/start` → согласие → 7 вопросов → результат → каждый из трёх продуктов.
2. Тестовая оплата ЮKassa (sandbox-магазин) → проверить, что покупатель
   получил инвайт в канал (практикум) или файл (книга), а `OWNER_CHAT_ID`
   получил уведомление об оплате.
3. M9 «Посмотреть другие варианты» не должно показывать уже купленное/
   забронированное.
4. Рестарт процесса бота посреди теста — прогресс должен сохраниться
   (проверка устойчивости чекпоинтов без FSM).
5. Напоминания: временно поставить `reminder_delay_hours=0` и
   `reminder_check_interval=30` через `/settings`, убедиться что R1–R6
   приходят по одному разу и не дублируются при повторных тиках scheduler'а.
6. `/settings` → сменить цену книги → она сразу видна в кнопке M8.1/M9 без
   рестарта процесса. Намеренно испортить `owner_chat_id` через `/settings`
   (например, вписать свой левый id) → доступ к `/settings` с исходного
   env-`OWNER_CHAT_ID` всё ещё работает.
